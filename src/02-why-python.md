# 为什么选择 Python

时间来到 2026 年，新兴编程语言可谓百家争鸣。Rust、Go、Zig、Kotlin……
它们各具特色，但都有一个共同的问题：学习门槛相对较高。

## Python

### 优点

* 生态成熟，不需要从零开始搭积木。
* 抽象层次高，无需过多关注底层细节。
* 语法简单，自描述性强，代码可读性高。
* 面向对象，并支持多种实现方式。[^1]

### 缺点

* 性能较差，即使将部分工作外包给其他语言，也会产生额外开销。

## 其他语言

* Rust 汲取了 C++ 内存模型的经验，糅合了多种经典编程语言所代表的编程范式。然而也正因如此，它拥有极为陡峭的学习曲线。
* Go 相对于 C 引入了许多抽象概念。它的类型系统设计也使得某些模式实现起来更加复杂。
* Zig 社区规模较小，标准库中的许多问题仍有待改进。其生态现状也存在一定风险：由于语言频繁引入不兼容变更，一些大型项目已经放弃 Zig，另一些项目则放弃跟进官方版本，转而长期维护自己的修改版 Zig 工具链。
* Kotlin 拥有大量语法糖，但它运行于 JVM[^2] 之上，因此在跨语言交互方面相对繁琐。

## 碎碎念

如果你了解“脚本”这一概念（Lua、JavaScript、Ruby、Visual Basic、Perl……），
可能会好奇这里为什么没有提到 Python 作为脚本引擎的用途。

原因在于工程上的取舍：Python 的标准库功能过于丰富，它更像是一门完整的编程语言。

用它来提供一个“脚本”环境，通常会走向两种可能：

1. 你写好的、别人写好的代码放进去无法运行，必须调整为适配精简或魔改环境的写法。
2. 这个脚本环境什么都能做，完全没有安全限制。换句话说，加载别人编写的地图脚本，甚至可能让你的电脑中病毒。

对于可能性 2，如果这个“脚本”并不面向用户（玩家）和社区开放，尚且可以接受；
但可能性 1 则会破坏 Python 最重要的优势之一——“生态成熟”，并同时失去完整标准库带来的价值。
我们不能说 MicroPython[^3] 是 Python 脚本。

《某某王国：世界》的游戏引擎便采用了魔改版 CPython，其脚本系统并不向玩家开放。

另一个小问题是，完整的 Python 至少需要约 30 MiB 的程序空间，这甚至已经比许多程序本身还要大。

这也导致在工程实践中，游戏引擎往往宁愿内嵌标准库功能羸弱、功能主要依赖具体软件通过脚本提供的 Lua，也不愿意使用 Python 作为脚本语言。

[^1]: Python 的子类判定方式存在多种特例，详见后续章节。
[^2]: Java 虚拟机，为 Java 的跨平台能力奠定了基础。虽然这使得与原生代码的交互变得十分痛苦，但也让 JVM 语言之间的互操作相对轻松。
[^3]: https://micropython.org/
